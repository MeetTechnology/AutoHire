# 01 Research Report

## 1. 需求背景

目标是承接“全球专家邮件邀约”后的专家端申请流程，不包含邮件发送前置环节。专家通过邮件链接访问网站，完成项目了解、简历初筛、补充信息、材料上传与提交。

## 2. 业务问题

当前流程的核心挑战不是页面数量，而是以下 4 个问题：

- 如何在固定主链接前提下准确识别专家身份
- 如何在专家中断后恢复进度
- 如何复用既有简历解析与资格初判能力
- 如何在大模型异步分析期间持续给用户反馈，避免无响应等待

## 3. 关键结论

### 3.1 身份识别

采用 `固定主链接 + 专家唯一 token` 最适合该场景。

- 邮件主链接固定，例如 `/apply`
- 每封邮件通过 query 参数附带唯一 `token`
- 后端用 `token` 绑定 `expert_id` 与 `application_id`
- 再次访问时通过该绑定恢复进度

### 3.2 进度恢复

必须以后端状态为准。

- 申请主表记录当前状态
- 文件上传、分析结果、缺失字段、补充信息都服务端持久化
- 前端根据后端快照恢复页面，而非依赖本地缓存

### 3.3 简历分析复用

复用已有项目的“简历分析能力”，不复用其业务页面。

- 本项目负责专家端流程编排
- 既有系统负责：
  - 简历解析
  - 字段提取
  - 大模型 + 固定提示词规则判断
  - 缺失字段识别

### 3.4 信息不足补填模式

缺失信息必须采用结构化字段补填，而不是自由文本补充。

- 后端返回 `missing_fields`
- 前端按配置动态渲染表单
- 补填后发起二次分析

## 4. 产品原则

- 让专家始终知道自己处于哪一步
- 大模型处理时必须有状态反馈
- 流程中断后能继续，而不是重来
- 判断结果必须清晰分为：
  - 不符合
  - 信息不足
  - 符合

## 5. 推荐交付方式

采用 Vibe Coding 常见结构化交付方式：

- 先完成 PRD
- 再完成技术设计和接口契约
- 然后形成 AI 代理约束与纵向切片实施计划
- 最后按测试计划逐步联调验证
