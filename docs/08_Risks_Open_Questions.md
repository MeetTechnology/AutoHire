# 08 Risks & Open Questions

## 1. 已确认决策

- 采用 `固定主链接 + 专家唯一 token`
- 资格判断使用“大模型 + 固定提示词规则”
- 缺失信息采用结构化字段补填
- 1-3 个工作日答复仅做页面提示
- 简历处理能力复用既有项目

## 2. 当前主要风险

### 2.1 token 被转发或泄露

风险：

- 他人获取链接后可能访问专家数据

建议：

- token 设置失效机制
- 关键页面增加二次身份确认能力作为预案

### 2.2 既有分析服务返回结构不稳定

风险：

- 前端动态表单无法稳定渲染

建议：

- 在适配层统一输出标准结构

### 2.3 大模型分析耗时过长

风险：

- 专家等待体验差

建议：

- 强制进入分析中页
- 明确展示处理中状态
- 支持失败重试

### 2.4 浏览器直传与本地测试环境稳定性

风险：

- OSS CORS、浏览器跨域策略与本地 dev server 时序会影响 E2E 稳定性

建议：

- 固定 Playwright 测试前置数据
- 统一本地测试端口与服务生命周期
- 将“服务端链路是否通过”和“浏览器端回归是否稳定”分开验证

### 2.5 材料上传量大

风险：

- 单次上传失败率升高

建议：

- 分类分批上传
- 单文件失败不影响其他文件

## 3. 仍待业务确认的问题

- token 是否需要有效期限制
- 已提交后是否允许后台触发补件
- 各材料分类是否存在后续的“推荐项/必填项”规则
- 页面是否需要明确展示当前步骤导航

## 4. 上线前检查清单

### 4.1 当前已完成

- token 鉴权、会话恢复与申请快照机制已在本地实现
- 状态恢复与多页面跳转已在本地 `mock` 模式验证通过
- 提交接口幂等已实现并已有自动化验证
- 简历分析适配层已建立，支持 `mock` 与 `live` 模式切换
- 文件上传已完成两步式接口与本地 mock 存储链路
- 真实 PostgreSQL 已完成建库、migration、seed 与关键流程落库验证
- 真实阿里云 OSS 已完成 upload intent、预签名 `PUT` 上传与材料回显验证
- 材料列表接口已收敛为安全摘要字段，不直接回传底层对象 key

### 4.2 当前待联调

- token 鉴权在真实邀约数据上的联调验证
- 简历分析适配接口在既有服务上的真实结构冻结
- 阿里云 OSS 的受控下载 / 受控预览验证
- 浏览器端直传回归稳定性验证

## 5. 当前进度判断

- 当前项目已完成 MVP 级别的本地实现，并已进入真实依赖联调收口阶段
- 当前主要风险已从“能否跑通流程”转为“浏览器端上传稳定性、受控访问链路、live 分析结构是否冻结”
- 下一阶段重点不再是补主流程页面，而是补齐 OSS 读链路、稳定 E2E、联调真实简历分析服务
