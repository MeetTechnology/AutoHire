# AI 补件功能实施检查清单

## 使用说明

- 本清单用于 AI 补件功能 Phase 1 的开发执行跟踪和 PR 验收复核。
- 勾选原则：只有在对应结果已可验证时，才可勾选。
- `建议默认` 项可按当前文档直接实现；`阻塞` 项只记录为待外部确认，不纳入当前完成定义。
- 本清单不替代原始 spec；阶段顺序以 `Development_Plan_AI_Material_Supplement.md` 为准，默认值与边界以 `Implementation_Decisions.md` 为准。

## 阶段 0：项目准备

- [x] 已形成本期实现默认值与阻塞项清单，开发可据此继续推进而不重写原始 spec。
  - 验证：`Implementation_Decisions.md` 中 `建议默认` 与 `阻塞` 已被实现方引用为本期决策入口。
- [x] 已建立本实施检查清单，且可直接用于开发跟踪与 PR 验收。
  - 验证：当前文档存在，覆盖阶段、回归项、非本期范围、外部阻塞项。
- [x] 已明确补件功能不影响现有 `/apply/materials` 主流程。
  - 验证：阶段 6 和“回归检查清单”中包含现有材料提交流程不受影响的检查项。

## 阶段 1：基础架构

- [x] 补件功能的类型与常量边界已建立，后续任务可基于统一定义继续开发。
  - 验证：存在补件领域类型、状态、类别、轮次或限制常量的统一定义，且能被前后端共享或稳定引用。
- [x] 补件错误码与响应模型已定义，API 与页面可按统一错误契约处理。
  - 验证：错误码、错误响应结构和成功响应骨架已形成，后续 API 不需要再临时发明错误格式。
- [x] 前端补件 client 骨架已建立，页面接入不依赖散落的请求逻辑。
  - 验证：补件页面使用的请求入口已集中封装，至少覆盖摘要、历史、上传、确认、同步等能力的占位接口。
- [x] 后端 material review client 骨架已建立，并保留 adapter 边界。
  - 验证：已区分 mock/live client 骨架，页面或 route 不直接耦合外部原始 payload。

## 阶段 2：数据库与数据模型

- [x] 补件审查相关 schema 与 migration 已落地，且不复用 `ApplicationMaterial` 存储补件文件。
  - 验证：数据库结构能独立表达补件请求、补件批次、补件文件、审查运行或历史数据，并体现“补件独立于现有材料”的边界。
- [x] 数据访问层已支持补件主流程读写。
  - 验证：存在稳定的数据访问接口，可完成摘要读取、历史读取、批次创建、文件确认、状态更新等核心操作。
- [x] 补件样例数据或构造能力已具备，后续 route test 与页面测试可直接复用。
  - 验证：测试或开发环境中可快速构造补件请求、补件文件、历史记录、latest 记录等示例数据。
- [x] 补件状态推导与 latest 维护逻辑已建立。
  - 验证：`materialSupplementStatus` 可由 latest run/latest request 推导；新版本请求通过新增记录表达，旧版本转为 history/latest=false，而不是原地覆盖；主视图默认隐藏已满足 latest request，history 仍保留可查。

## 阶段 3：核心 API

- [ ] 补件访问控制与 service 骨架已实现，且非 `SUBMITTED` application 无法进入补件流程。
  - 验证：服务层或 route 层对 application 状态有限制；非 `SUBMITTED` 请求会返回明确拒绝结果。
- [ ] 摘要与首轮审查 API 已实现，提交完成页可读取补件摘要并触发首轮逻辑。
  - 验证：存在可供提交完成页调用的摘要/首轮触发接口，能返回补件入口所需的最小信息集。
- [x] 补件快照与历史 API 已实现，历史数据默认可查，且主页面默认隐藏已满足请求。
  - 验证：主页面读取与 history 页面读取均有 API 支撑，latest 与 history 结果可区分。
- [x] 补件上传批次与上传凭证 API 已实现，并沿用现有 upload 规则。
  - 验证：文件扩展名限制沿用当前 allowlist；大小限制沿用 `20 MB`/`100 MB`；每类每批次默认最多 `10` 个文件。
- [x] 补件文件确认、删除和批次确认 API 已实现，并支持按类别完成一次补件提交。
  - 验证：文件在确认前可删除；批次确认后可形成该类别的一次有效补件输入。
- [x] 审查运行状态与显式 `sync` API 已实现，mock 优先路径可闭环。
  - 验证：系统可通过手动或服务端触发 `sync` 拉取结果；不依赖 live callback 也能完成 Phase 1 演示与测试。
- [x] 内部 callback API 已保留集成接口位，但未写死未确认的 live 协议细节。
  - 验证：route 或 service 入口存在；未把未确认的 header、签名算法、payload 结构固化为最终契约。

## 阶段 4：核心页面

- [x] 提交完成页已接入补件摘要与首轮触发入口，但不改变现有材料提交完成主路径。
  - 验证：提交完成后可看到补件摘要或入口；现有完成页核心行为不被替换或阻断。
- [x] `/apply/supplement` 主页面已实现，作为补件主入口。
  - 验证：页面路径采用 `/apply/supplement`，并展示补件类别、待处理项、已满足项隐藏规则或当前轮次信息。
- [x] 补件文件选择、去重与批次确认交互已实现。
  - 验证：同名且同大小文件会被识别为重复；用户可完成选择、删除、确认批次的完整交互。
- [x] `/apply/supplement/history` 页面已实现，并作为独立历史页而非 modal。
  - 验证：页面路径采用 `/apply/supplement/history`，可查询历史补件请求、历史审查结果或旧版本记录。
- [x] 访问失效、无权限和空状态已补齐。
  - 验证：application 不存在、状态不允许、无待补件项、历史为空等情况均有明确页面反馈。

## 阶段 5：AI 能力接入

- [x] material review mock client 已实现，可支撑本地开发、联调和测试。
  - 验证：在无 live backend 的情况下，系统可生成可预期的补件审查状态或结果样例。
- [x] material review live client 已实现骨架，并通过 adapter 与业务逻辑解耦。
  - 验证：live client 有独立入口与配置边界，但页面和 route 不直接依赖外部原始字段。
- [x] 审查结果 adapter 与入库事务已实现。
  - 验证：外部或 mock 返回结果先映射到内部最小结构，再完成持久化；UI 读取的是内部结构而非原始 payload。
- [x] 同步策略已优先支持 mock + 显式 `sync`，callback 仅保留未来接入路径。
  - 验证：当前功能闭环不依赖 callback；callback 作为预留集成面存在但不构成当前完成阻塞。

## 阶段 6：测试与修复

- [x] 补件数据与 service 单元测试已覆盖核心状态与规则。
  - 验证：至少覆盖类别范围、轮次上限、批次上限、latest/history 维护、非 `SUBMITTED` 拒绝等场景。
- [x] 补件 API route tests 已覆盖核心接口。
  - 验证：至少覆盖摘要、历史、上传、确认、删除、批次确认、同步、权限拒绝等 API 场景。
- [x] 补件页面组件与交互测试已覆盖关键用户操作。
  - 验证：至少覆盖补件主页面渲染、文件选择、去重、确认、历史页展示、异常状态提示等场景。
- [ ] 补件 E2E 流程测试已覆盖主链路。
  - 验证：至少覆盖提交后进入补件、上传补件、确认批次、查看结果或历史的端到端流程。
- [ ] 现有材料提交流程回归问题已修复或确认不存在。
  - 验证：`/apply/materials` 现有进入、上传、删除、确认、提交流程仍可正常完成。

## 阶段 7：上线准备

- [ ] 补件相关环境变量与部署文档已补齐。
  - 验证：mock/live 所需配置项、内部 callback 预留配置项和运行说明已记录到部署文档或环境变量说明中。
- [ ] 上线前数据迁移与回滚检查已完成。
  - 验证：migration 可执行，回滚策略或恢复方案有记录，且不会误伤现有材料数据。
- [ ] 最终 QA 清单与发布记录已准备完成。
  - 验证：发布前检查项可供 QA 或 reviewer 逐项确认，且包含本期边界与回归结论。

## 回归检查清单

- [ ] 现有 `/apply/materials` 提交流程仍可进入、上传、删除并最终提交。
  - 验证：不经过补件流程时，原有材料提交链路行为与当前线上/基线行为一致。
- [ ] AI 补件流程为提交后独立体验，未并入现有 `/apply/materials` 页面。
  - 验证：补件主入口位于提交完成后的补件路径，主材料页未被改造成补件承载页。
- [ ] 非 `SUBMITTED` application 不会进入补件流程。
  - 验证：草稿、处理中或其他非允许状态访问补件接口/页面时会被拒绝或重定向。
- [ ] 补件文件存储与管理独立于 `ApplicationMaterial`。
  - 验证：补件文件模型、查询路径和历史记录不依赖把补件文件混入现有材料表。
- [ ] 已满足的补件请求默认不在主页面堆叠展示，但可在历史中查询。
  - 验证：主页面与历史页面展示策略符合该默认决策。

    - Multi-language support
